# Travel Planner

> **A note on AI usage.** The implementation, architecture and trade-offs are
> my own. I used Claude Code to polish my `docker-compose.yml` (healthchecks
> and `depends_on: condition: service_healthy` for the API service) and to
> help structure this README so the setup and endpoint documentation reads
> cleanly.

Small CRUD API for managing travel projects and the places a traveller wants to
visit. Places are validated against the
[Art Institute of Chicago API](https://api.artic.edu/docs/#collections) — the
test task treats artworks as "places".

## Stack

- FastAPI + Uvicorn
- PostgreSQL (via SQLAlchemy 2.x async + asyncpg)
- Redis for caching third-party responses
- httpx as the HTTP client

## Quick start (Docker)

```bash
cp .env.example .env   # optional, defaults work with compose
docker compose up --build
```

API will be available at `http://localhost:8000`, interactive docs at
`http://localhost:8000/docs`.

## Local run (without Docker)

You need a running Postgres and Redis. Point `DATABASE_URL` / `REDIS_URL` at
them in `.env` and then:

```bash
python -m venv .venv
source .venv/bin/activate        # on Windows: .venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

Tables are created automatically on app startup (`metadata.create_all`), which
is enough for a test task. In a real project I'd use Alembic.

## Environment variables

| variable                | default                                                              | description                                 |
|-------------------------|----------------------------------------------------------------------|---------------------------------------------|
| `DATABASE_URL`          | `postgresql+asyncpg://postgres:postgres@localhost:5432/travel`       | async SQLAlchemy URL                        |
| `REDIS_URL`             | `redis://localhost:6379/0`                                           | Redis connection URL                        |
| `ARTIC_BASE_URL`        | `https://api.artic.edu/api/v1`                                       | Art Institute API base URL                  |
| `ARTIC_CACHE_TTL`       | `3600`                                                               | Cache TTL for artwork lookups, in seconds   |
| `BASIC_AUTH_USERNAME`   | `admin`                                                              | Username for HTTP Basic Auth                |
| `BASIC_AUTH_PASSWORD`   | `admin`                                                              | Password for HTTP Basic Auth                |

## Authentication

All `/projects` and `/projects/{id}/places` endpoints are protected with HTTP
Basic Auth. Credentials come from `BASIC_AUTH_USERNAME` / `BASIC_AUTH_PASSWORD`
(defaults: `admin` / `admin`). `/health` and the `/docs` Swagger UI are left
open so the Docker healthcheck and API exploration still work.

```bash
curl -u admin:admin http://localhost:8000/projects
```

## Endpoints

Full spec is auto-generated by FastAPI: open `/docs` (Swagger UI) or
`/redoc` after the app is up. Short summary:

**Projects**
- `POST   /projects` — create a project, optionally with a list of places
- `GET    /projects` — list with `limit`, `offset`, `is_completed`, `name` filters
- `GET    /projects/{id}` — single project with its places
- `PATCH  /projects/{id}` — update name / description / start_date
- `DELETE /projects/{id}` — delete (rejected with 409 if any place is visited)

**Places**
- `POST   /projects/{id}/places` — add a place (validated against artic.edu)
- `GET    /projects/{id}/places` — list places in a project
- `GET    /projects/{id}/places/{place_id}` — single place
- `PATCH  /projects/{id}/places/{place_id}` — update `notes` or `is_visited`

When every place in a project is marked `is_visited=true`, the project is
automatically flipped to `is_completed=true`.

## Example requests

Create a project with two places in one shot:

```bash
curl -u admin:admin -X POST http://localhost:8000/projects \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Chicago weekend",
    "description": "short trip",
    "start_date": "2026-05-01",
    "places": [
      {"external_id": 129884, "notes": "must see"},
      {"external_id": 27992}
    ]
  }'
```

Add a place to an existing project:

```bash
curl -X POST http://localhost:8000/projects/1/places \
  -H "Content-Type: application/json" \
  -d '{"external_id": 16568}'
```

Mark a place as visited:

```bash
curl -u admin:admin -X PATCH http://localhost:8000/projects/1/places/1 \
  -H "Content-Type: application/json" \
  -d '{"is_visited": true}'
```

## Notes / trade-offs

- `metadata.create_all` is used instead of migrations to keep the test setup
  minimal. For anything long-lived, swap it for Alembic.
- Caching is intentionally simple: every artwork lookup hits Redis first,
  successful responses are cached for `ARTIC_CACHE_TTL`, and 404s are cached
  briefly (max 5 minutes) to avoid hammering the API on bad IDs.
- Per-project place limit is enforced both at the application level (nicer
  error message) and through a unique constraint on `(project_id, external_id)`
  in Postgres.
